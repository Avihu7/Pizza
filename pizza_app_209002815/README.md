## הרצה - צד שרת
```bash
cd server
npm install
npm start
```
השרת רץ כברירת מחדל על פורט `3001` (ניתן לשנות עם משתנה סביבה `PORT`).

## הרצה - צד לקוח
```bash
cd client
npm install
npm run dev
```
הלקוח רץ כברירת מחדל על `http://localhost:5173` ופונה לשרת בכתובת `http://localhost:3001`
(ניתן לשנות עם משתנה סביבה `VITE_API_URL`). יש להריץ את השרת במקביל כדי שהלקוח יעבוד.

לבנייה לפרודקשן: `npm run build` בתיקיית `client`.

## איפה מחושב המחיר, ולמה
המחיר הכולל מחושב **אך ורק בשרת**, בקובץ `server/src/services/pricing.js`, על בסיס
מזהי הפיצה/גודל/תוספות שהתקבלו מהלקוח, מול נתוני התפריט האמיתיים שנמצאים ב-
`server/src/data/menu.js`. הלקוח שולח רק מזהים (pizzaId, sizeId, toppingIds),
ולעולם לא שולח מחיר. הלקוח מציג "מחיר משוער" מקומי כדי לשפר את חוויית המשתמש לפני
השליחה, אך המחיר הסופי שמוצג למשתמש לאחר יצירת ההזמנה (`totalPrice`) הוא תמיד זה
שחוזר מהשרת. כך נמנע מצב שבו לקוח (או באג בצד הלקוח) "משנה" מחיר.

## תשלום מדומה
התשלום המדומה מתבצע בצד הלקוח לפני שליחת ההזמנה לשרת (לחיצה על כפתור
"בצע תשלום מדומה ושלח הזמנה"). לכן, כאשר הזמנה מגיעה לשרת היא כבר "שולמה" מבחינת
הזרימה הלוגית של המערכת, ולכן `paymentStatus` מתחיל תמיד כ-`"paid"` (ולא
`"pending"`), ולא נשמר שום פרט תשלום אמיתי - זהו תשלום מדומה בלבד.

## מה קורה כאשר לקוח שולח הזמנה לא תקינה
השרת מריץ סדרת ולידציות (`server/src/validation/orderValidation.js`) על גוף
הבקשה: שדות חובה חסרים, פחות מפיצה אחת, sizeId/pizzaId/toppingId שלא קיימים
בתפריט, יותר מ-3 תוספות לפיצה, או יותר מ-5 פיצות בהזמנה (הכלל האישי) - כל אחד
מהמקרים האלה גורם לתשובת `400` עם הודעת שגיאה ברורה בשדה `error`, וההזמנה כלל
לא נשמרת. הלקוח מציג את הודעת השגיאה למשתמש מתחת לכפתור השליחה.

## שינויים מהתכנון של תרגיל 1
לא בוצעו שינויים מהותיים בתכנון. השינוי היחיד הוא התאמה טכנית: הוספת מזהה (`id`)
ייחודי לכל פריט תפריט (פיצה/גודל/תוספת), כדי שהלקוח יוכל להתייחס לפריטים לפי מזהה
ולא לפי שם - הכרחי למימוש ה-API אך לא משנה את מבנה הנתונים או הלוגיקה העסקית
שתוכננה.

## שאלות
**1. מה ההבדל בין צד הלקוח לצד השרת במערכת שלכם?**
צד השרת הוא מקור האמת: הוא מחזיק את התפריט, מאמת כל קלט, מחשב מחיר, שומר הזמנות
בזיכרון ואוכף את מנוע מצבי ההזמנה. צד הלקוח הוא רק ממשק - הוא מציג מידע שמגיע
מהשרת ושולח בקשות, ולא מקבל אף החלטה עסקית בעצמו (אפילו לא לגבי המחיר).

**2. איפה מחושב המחיר הכולל ולמה?**
בשרת בלבד, ב-`server/src/services/pricing.js`, כדי שלא ניתן יהיה לזייף מחיר
מהדפדפן - השרת מחשב מחדש מהתפריט האמיתי בכל בקשה.

**3. מה קורה כאשר לקוח שולח הזמנה לא תקינה?**
השרת מחזיר `400` עם הודעת שגיאה מתארת (למשל מזהה לא קיים, יותר מ-3 תוספות, או
יותר מ-5 פיצות), וההזמנה לא נוצרת ולא נשמרת.

**4. מה קורה לאחר שהתשלום המדומה מצליח?**
הלקוח שולח את ההזמנה לשרת (`POST /api/orders`). השרת מאמת, מחשב מחיר, יוצר הזמנה
עם `status: "new"` ו-`paymentStatus: "paid"`, ומחזיר אותה עם קוד `201`. הלקוח מציג
אישור עם מספר ההזמנה והמצב הנוכחי.

**5. מהו הכלל האישי שחל עליכם?**
אי אפשר להזמין יותר מ-5 פיצות באותה הזמנה (ספרה אחרונה בת"ז 209002815 היא 5).

**6. מה היה החלק הכי מאתגר בתרגיל?**
לתכנן מנוע מצבים נוקשה (new → preparing → ready → delivered) שאוכף גם מניעת דילוג
שלבים וגם מניעת חזרה אחורה, ולהחזיר קוד השגיאה הנכון (409) בלי לפגוע בזרימה
התקינה.

**7. איזו החלטת תכנון אחת קיבלתם ולמה?**
בחרנו לתת לכל פריט תפריט מזהה (`id`) ייחודי במקום להסתמך על שם הפריט, כדי שהלקוח
תמיד יתקשר עם השרת דרך מזהים יציבים - כך שינוי עתידי בשם תצוגה לא ישבור הזמנות
קיימות או ולידציה.
